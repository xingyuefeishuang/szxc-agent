# 订单模块锁分层约定实施计划

## 背景
- 当前订单模块同时存在内部 Controller 入口和外部渠道 SPI / 适配层入口。
- 若在核心 Service 普遍补请求级分布式锁，容易与入口层锁重复，并导致锁粒度混杂、顺序不清。
- 抖音等外部链路还存在“一次入口请求调用两个核心方法”的场景，更需要先明确锁分层约定。

## 目标
- 统一订单模块的请求级锁放置层次，避免后续在核心层重复加锁。

## 约定
1. 内部单请求级分布式锁统一收敛在 Controller 入口层。
2. 外部渠道请求级分布式锁统一收敛在 SPI / 适配层。
3. 核心 Service 不承担请求级锁职责，只负责：
   - 业务命令执行
   - 状态机流转
   - 业务幂等判断
4. 若后续补资源锁，应单独按“订单聚合资源锁”设计，不与请求级锁混用。

## 本次落地
1. 在 `OrderController` 关键入口补充锁分层注释。
2. 在 `DouyinChannelAdapter.handleCreateOrder(...)` 补充外部锁分层注释。
3. 在 `OrderServiceImpl.createOrder(...)` 补充核心层不承接请求级锁的备注。
