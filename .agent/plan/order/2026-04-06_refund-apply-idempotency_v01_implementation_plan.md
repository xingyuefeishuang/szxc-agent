# 退款审核幂等修正实施计划

## 背景
- 抖音退款审核当前使用 `get -> 处理 -> set` 的 Redis 缓存式幂等。
- 该写法无法阻止并发重复请求同时穿透到 `refundApplyService.applyRefund(...)`。
- 核心层当前也没有基于 `bizUniqKey` 的数据库唯一兜底，因此短期需先把适配层幂等改成抢占式。

## 实施方案
1. 保留现有 `bizUniqKey` 作为退款审核幂等键。
2. 增加“处理中”占位值，进入业务前先用 `setIfAbsent` 抢占该幂等键。
3. 如果未抢占成功：
   - 命中最终结果缓存则直接返回最终审核结果；
   - 仍处于处理中则返回重试响应，不再重复受理。
4. 业务成功后把幂等键改写为最终审核结果缓存。
5. 业务异常时，仅当前持有者清理“处理中”占位，允许后续重试重新进入。

## 边界说明
- 本次不改数据库结构，不新增退款申请幂等表。
- 本次不调整退款结果通知逻辑，先聚焦退款审核入口。
- 长期仍建议补数据库级外部幂等记录，否则 Redis 丢失后无法作为最终真相。
