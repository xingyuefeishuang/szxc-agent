# 退款结果通知幂等修正实施计划

## 背景
- 抖音退款结果通知当前通过 `hasKey` 直接短路。
- 该写法无法阻止并发重复通知同时进入核心退款完成逻辑。
- 若通知处理中被直接视为“已完成”，还可能在前一个线程异常退出时吞掉后续重试。

## 实施方案
1. 保留现有 `bizUniqKey` 作为退款通知幂等键。
2. 在处理通知前先读取该幂等键：
   - 命中最终完成标记时直接返回成功；
   - 命中处理中占位时返回重试。
3. 若缓存不存在，则用 `setIfAbsent` 抢占处理权。
4. 抢占成功后执行 `completeRefundByOrderNo(...)`。
5. 处理成功后写入最终完成标记。
6. 处理失败或异常时，仅当前持有者清理处理中占位，允许后续通知继续重试。

## 边界说明
- 本次不改数据库结构。
- 退款通知接口仍以核心层 `completeRefundByOrderNo(...)` 的状态机兜底为最终业务保障。
- Redis 仍然只是前置幂等加速层，不是最终真相源。
