# 门票直连接入手册

**更新时间**：2026-03-09

## 一、 文档简介
本文介绍使用美团门票直连技术商接入的注意事项、使用流程和操作步骤。
*   **适用对象**：2025年11月之后，通过美团技术服务合作中心完成入驻的服务商。
*   **注意事项**：在此之前通过美团门票开放平台、本地生活开放平台接入的开发者，请参考门票业务接口接入手册，切勿参考本文直接修改任何对接参数。

### 1.1 注意事项
*   **接入前提**：美团门票直连仅限符合条件的开发者接入。开发者已经或计划和门票供应商（下称景区）达成合作，需要通过开发接口来接入美团，实现订单和产品纬度的直连对接。
*   **资源提供**：本平台不负责提供合作资源，接入本平台并不代表美团会提供任何资源合作。美团会审核入驻门票业务线开发者的合作情况，开发者需自行与景区达成合作后，再在本平台完成入驻对接。如有审核未通过等问题，请联系合作景区负责的美团商务同学内部提报对接需求。
*   **违规处罚**：如开发者有违规使用平台接口及数据的行为，开放平台有权给予停用对应应用、停用测试权限、停用开发者账号等处罚。如开发者的违规行为给用户、美团或景区造成损害，或导致用户、美团或景区遭受第三方权利主张、监管处罚的，违规开发者应赔偿用户、美团和景区全部损失（含直接经济损失和各项合理费用）。
*   **参考规范**：美团服务商系统安全规范、第三方应用开发安全规范、违规行为管理规范

## 二、 整体接入流程
整体接入流程主要分为五个部分：
1.  **入驻流程**：开发者需要前往美团技术服务合作中心完成注册并入驻。请按照三方服务商身份完成入驻，切勿直接使用景区账号登陆。后续创建应用和维护接口都需要通过账号登陆，务必以技术开发者信息自行注册并入驻。完成入驻后联系美团商务同学周知内部审批。审批通过后点击开通业务，选择门票直连，并按平台要求完成合同签署。
2.  **开发流程**：开发者需要根据美团门票API完成接口开发。开发手册详见下文。
3.  **配置流程**：开发者需在控制台完成开发者创建并获取接口授权。等待审批通过后，配置验收环境地址。
4.  **测试流程**：开发者需自助进行API用例测试。用例测试通过需要进行门店测试（自动化测试），自主配置测试产品并进行订单测试及失败文案测试。测试完成后点击提交测试报告，等待美团审批。
5.  **上线流程**：测试审批通过后，配置生产环境地址并等待审批，应用状态为已上线后，联系美团业务同学获取PartnerId，其余对接参数可在正式环境配置页查看。配置后可上线正式产品售卖。

## 三、 开发者入驻
1.  注册并登录美团技术服务合作中心。
2.  **身份要求**：请用开发者信息，以三方服务商身份完成入驻，切勿直接使用景区账号登录。后续创建应用和维护接口都需要通过账号登录，务必以技术开发者身份自行注册并完成入驻。
3.  **审批流程**：完成入驻后联系美团商务同学周知内部审批。
    *   **PS**：纯门票技术服务商，建议仅勾选门票直连/门票。如有交叉业务，需按照较严格的业务入驻门槛执行。

## 四、 开发说明
开发者需围绕美团门票API完成接口开发。
*   **接口分类**：其类目下分为【产品API】【交易API】【辅助API】三大类接口。
*   **安全认证**：为保证数据安全，商家同美团的接口交互需要满足美团BA认证。
*   **请求方式**：技术合作中心提供的业务API，目前只支持HTTP POST一种请求方式。
*   **Content-Type**：指定Body Content-Type为：application/x-www-form-urlencoded
*   **参数传递**：Post请求时，所有参数均通过表单传递，请勿将请求参数放到Query参数中。

| sign | String | | 是 | 数字签名 | ceb760af62431e20fd2e4aa1bbf21d2c17012dc0 |
| timestamp | Long | | 是 | 时间戳，单位秒 | 1577771730 |
| charset | String | | 是 | 字符编码 | utf-8 |
| developerId | Long | | 是 | 服务商身份标识 | 123123 |
| version | String | | 是 | 版本号，固定传2 | 2 |
| businessId | int | | 是 | 业务类型id，门票业务固定传66 | 66 |
| biz | String | | 是 | 业务请求参数 | {"ticket":"10346762252","number":2} |
| appAuthToken | String | | 否 | 门店令牌，接口需要授权时必填 | abc |

> **签名规则说明文档**：[https://developer.meituan.com/docs/biz/comm-dev-isv-sign-rule](https://developer.meituan.com/docs/biz/comm-dev-isv-sign-rule)

### 4.2 签名计算方式
以执行核销通知接口`/rhon/mtp/api/order/consume/notice`为例，假如开发者的signKey是k4ke1u0r44bgq1kx，请求的参数如下：
*   charset：UTF-8
*   timestamp：1761737557
*   developerId：117092
*   biz: `{"partnerId":26465,"body":{"orderId":5008017280866233593,"partnerOrderId":"partnerOrderId-5008017280866233593","quantity":1,"usedQuantity":1,"refundedQuantity":0,"voucherList":[{"voucher":"voucher-5008017280866233593","voucherPics":"www.meituan.com","voucherInvalidTime":"2025-10-29 19:00:00","quantity":1,"status":1}]}}`
*   version: 2
*   businessId: 66

**步骤**：
1.  **参数排序与拼接**：将请求参数中除 sign 外的多个键值对，根据键按照字典序排序，并按照 "key1value1key2value2..." 的格式拼成一个字符串。
    *   结果为：`biz{"partnerId":26465,"body":{"orderId":5008017280866233593,"partnerOrderId":"partnerOrderId-5008017280866233593","quantity":1,"usedQuantity":1,"refundedQuantity":0,"voucherList":[{"voucher":"voucher-5008017280866233593","voucherPics":"www.baidu.com","voucherInvalidTime":"2025-10-29 19:00:00","quantity":1,"status":1}]}}businessId66charsetutf-8developerId 117092timestamp 1761737557version2`
2.  **拼接密钥**：将 signKey 拼接在 1 中排序后的字符串前面得到待签名字符串：
    *   `k4ke1u0r44bgq1kxbiz{"partnerId":26465,"body":{"orderId":5008017280866233593,"partnerOrderId":"partnerOrderId-5008017280866233593","quantity":1,"usedQuantity":1,"refundedQuantity":0,"voucherList":[{"voucher":"voucher-5008017280866233593","voucherPics":"www.baidu.com","voucherInvalidTime":"2025-10-29 19:00:00","quantity":1,"status":1}]}}businessId66charsetutf-8developerId 117092timestamp 1761737557version2`
3.  **SHA1加密**：使用 sha1 算法加密待加密字符串并转为小写即为 sign：
    *   `37350dbf19b73248c9d440ea13cad18b13ccada7`
4.  **AppAuthToken**：接口需要授权时必填，直连门票业务所有接口无需授权，故该字段无需填写。

### 4.3 【产品API】说明
*   **背景**：为降低景区系统商开发难度，提升对接效率，美团当前景区、产品相关信息（如退款规则、游玩人信息填写规则，景区开闭园时间等）都通过商务手工上单，只需要接入价格日历库存相关接口即可。
*   **开放接口**：
    *   **拉取价格日历**：普通价格日历产品在美团后台录入价格、库存等信息。
    *   **拉取多层价格日历V2**：分时预约产品在美团后台录入或变更价格、库存、场次及区域等信息。
    *   **多层价格日历变化通知V2**：分时预约产品在美团后台录入或变更价格、库存、场次及区域等信息。
*   **总结**：
    *   产品API 拉取价格日历（普通门票价格日历库存，正常对接）
    *   拉取多层价格日历V2（分时预约门票价格日历库存，正常对接）
    *   多层价格日历变化通知V2（分时预约门票价格日历库存，正常对接）

### 4.4 【交易API】说明
该部分为游客下单、出票、核销、退款、改签等接口汇总，根据景区实际业务规则进行接口对接。
*   **必接接口**：订单创建V2、订单出票、订单查询、订单退款、已退款消息、订单关闭消息、订单消费通知7个为必接接口，若缺少其中部分接口对接则不予上线。
*   **已退款消息接口说明**：该接口并非退款请求接口，美团侧调用该接口时，代表订单已经给用户退款，通知景区撤销核销凭证，同步修改订单状态为已退款。

**个性化接入说明**：
*   **同步/异步出票**：
    *   同步出票：只对接订单出票接口即可，美团通过该接口请求后，景区直接返回出票结果。
    *   异步出票：需要对接订单出票和订单出票通知2个接口，由订单出票接口返回出票中状态，后续景区通过订单出票通知接口主动通知美团出票最终结果。
*   **同步/异步退款**：
    *   同步退款：只对接订单退款接口即可，美团通过该接口请求后，景区直接返回退款结果。
    *   异步退款：需要对接订单退款和订单退款通知2个接口，由订单退款接口返回退款审批状态，后续景区通过订单退款通知接口主动通知美团退款最终结果。
*   **改签接口**：根据实际业务需求选择性接入即可，目前支持游玩人改签与日期改签。

### 4.5 【实名制下单】与【实名制核销、退款】说明（重要）
*   **实名制下单**：游玩人传参根据景区需求在上单后台勾选即可（支持N张票传1个或N个游玩人信息，也支持1张票需要传N个游玩人信息），传参信息一般为姓名、手机号、证件类型、证件号。
*   **实名制核销、退款（重要）**：
    *   美团口径内【实名制下单≠实名制订单】。
    *   实名制下单按照景区规则传对应游玩人信息即可。
    *   **判定标准**：需要景区技术商对接订单出票（或订单出票通知）的`realNameType`字段。
        *   若该字段传0（或不传默认为0）时为非实名制订单，退款和核销无需传身份证信息。
        *   若该字段传1时，出票（出票通知）、核销通知、退款（退款通知）接口必传`credentialList`下相关参数，退款和核销需要凭码+身份证信息（即需要券码和游客信息相匹配）。
    *   **风险提示**：若`realNameType`与`credentialList`传参不符，如前者传1，后者未传，对应产品会被美团惩罚下线，需格外注意！
    *   **场景对比**：
        *   实名制核销退款（如高铁票）：票码A对应张三，票码B对应李四。张三只可以刷票码A，无法刷票码B入园。退款时，也需要选择游玩人退款。
        *   非实名制核销退款：按照数量来。一个订单买3张票，可以三个人都入园，也可以一个人入园三次。退一张票后，3个人可以任选2个人游玩。
    *   **注意**：刷身份证件入园不一定就是实名制核销退款，需要按照系统对接的标准来判定。

### 4.6 【一码多验】与【一码一验】说明（重要）
*   **定义**：
    *   一码一验：1个订单内，N张票对应N个凭证。
    *   一码多验：1个订单内，N张票共用1个凭证。
*   **传参规则**：
    *   `voucherType`为0时：代表此订单是一码多验或无需凭证。由于只有唯一1个凭证或无凭证，则在核销、退款时以该订单数量为主，美团不会返回凭证给商家。
    *   `voucherType`为1时：代表此订单是一码一验。景区需要传对应票数的凭证给到美团，在出票、核销和退款环节，美团与景区之间接口交互会对应到具体凭证维度，此时双方需要在`voucherList`字段传对应核销或退款的二维码。
*   **注意**：在非实名制核销退款的情况下，美团退款时请求的凭证码是随机选择未使用的，如果景区存在延迟核销或定时核销等情况，需要针对美团请求退款的凭证码做校验处理。

### 4.7 普通/实名制核销、退款&传码方式字段说明（重要）
实名制核销和一码一验、一码多验并不冲突，具体传参如下：

| 核销、退款 | 出票 | 出票 | 核销、退款 | 出票 | 出票、核销、退款 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **普通or 实名制** | **一码多验（或无凭证）or一码一验** | **voucherType** | **voucherList** | **realNameType** | **credentialList** |
| 普通 | 一码多验（或无凭证） | 0 | null或1个凭证码 | 0 | null |
| 普通 | 一码一验 | 1 | 传对应凭证 | 0 | null |
| 实名制 | 一码多验 | 0 | 1个凭证码 | 1 | 传证件与凭证对应关系，多个证件可对应同一个凭证码 |
| 实名制 | 一码一验 | 1 | 传对应凭证 | 1 | 传证件与凭证对应关系 |

### 4.9 接口总说明
1.  **Code定义**：code200是美团接口交互唯一“成功”标识。除了“出票接口返回598代表【出票中】、退款接口返回602代表【退款审批中】、退款接口返回601代表【退款成功】”之外，一切非200的code均表示非成功状态。
2.  **错误码**：各个接口都有约定的错误码，请严格按照约定的错误码返回code值，不要自造，否则系统会直接判定商家接口异常，影响交易。
3.  **接口文档**：接口说明以接口文档为准，请务必注意各种type、status的代表含义，因为美团会针对不同的值进行不同的自动化处理，详见接口验收标准的表格说明。
4.  **出票接口**：下单之后的“订单出票接口”200成功表示着用户能入园、订单已经出票了。一切要向上游二次出票的订单一定先返回598【出票中】，再调用我们的出票通知接口告知出票结果，否则用户投诉虚假出票会拉黑上单权限。
5.  **幂等校验**：美团用同一订单号/退款流水号，多次请求你方时，你们一直返回的是实际订单状态，而不是因为多次请求而拒绝我们。如订单A，在你方订单状态是【下单成功】，我们用订单A多次请求你方下单接口，请一直返回【下单成功】，且返回字段完全符合我方接口要求，即和接口文档上该接口约定字段一致。
6.  **美团订单号位数兼容性升级**：因美团交易平台化重新设计了统一的订单ID生存规则：long数据类型，占63个bit，十进制19位。目前度假线上仅为15位（十进制）。因此要求系统商（包含跟美团对对接的所有版本）支持不同位数（15-19位）订单号已有的传递、存储、展示逻辑。如果无法兼容，会出现下单失败，订单状态不一致等问题，造成财务账差。
7.  **接口地址白名单**：如果您的地址是带端口号的ip地址，请注意，美团只支持下面的端口号，除此之外的不支持，可用的端口号HTTP:80,HTTPS:443。

## 五、 配置说明

### 5.1 名词解释
*   **开发者**：商家用于对接美团门票开放平台的API集合，每个开发者ID对应两套环境的地址集。
*   **验收环境**：验收环境特指商家测试环境，用于自动化测试。
*   **生产环境**：生产环境特指商家生产环境，测试通过后可以配置并申请上线。

### 5.2 开通业务
1.  进入控制台，搜索并选择开通门票直连业务，完成相关协议签署。
    *   **PS**：纯门票技术服务商，建议仅开通门票直连。如有交叉业务，需按照较严格的业务入驻门槛执行。

### 5.3 开发者申请
1.  点击申请开发者，选择其他-门票直连，勾选默认场景，选择应用接入。
2.  **产品名称**：建议命名为：系统名称-景区名称。
3.  **接口授权**：默认勾选全部，可按需自行对接。

### 5.4 开发者接口配置
1.  开发者申请成功后，会默认开通测试环境及正式环境两套接口。
2.  可在开发者列表，点击测试开发者，配置回调地址（美团请求景区）。
3.  **注意**：正式地址需在测试开发者通过自动化测试后方可配置上线。
4.  **重要提示**：门票直连无需设置技术合作中心消息回调地址。

## 六、 测试说明

### 6.1 名词解释
*   **自动化测试**：实际业务测试，需要开发者自行按照指引配置测试产品，用美团APP扫码下单并校验，测试partnerId务必使用26465，调用美团生产环境。

### 6.2 自动化测试

点击控制台-门票直连对接，新增产品。不同产品类型说明如下。

## FAQ

### 1. 入驻账号只能用技术人员的吗？能否用景区的账号入驻？
*   **A**：因为本平台使用者为开发人员，建议系统服务提供方技术人员入驻，便于后续使用及变更。

### 2. 如果注册者离职，账号该如何继承？
*   **A**：建议记录好账号密码，离职交接时换绑手机号。如果无法联系到注册者，可以联系美团同学帮助。

### 3. 新建或编辑测试产品后保存提交时，提示系统异常？
*   **A**：
    *   **新建产品**：测试产品至少需要一个游玩人的手机号，请勾选需要游玩人信息——手机号。
    *   **编辑产品**：美团针对测试产品会有巡检机制，如果修改产品后提交报错，请删除产品重新上测试产品。如果仍有问题，联系美团业务同学提交内部排查。

### 4. 新建测试产品时，提示此单已存在？
*   **A**：产品编码过于简单导致重复，建议更换复杂编码（partnerDealId）。

### 5. 为什么产品审核时间突然变长，提交产品一直提示在审核中？
*   **A**：
    *   正常产品审核时间为3-5分钟，如果超过这个时间刷新还是显示产品在审核，就是进入到费率审核流程，需要联系美团商务同事内部催审。
    *   为避免进入费率审核，请务必按照页面要求的美团价=门市价=0.02，结算价=0.01配置，且不要随意变更价格。

### 6. 为什么下单失败后，显示产品已约满？
*   **A**：部分报错会触发美团自动化策略导致产品下线，需要去测试产品配置页面手动恢复产品上线，如果手动上线失败，请在对接群内联系美团商务同事提报工单处理（需提供商家产品编码）。

### 7. 为什么拉取价格日历后，日志没有内容？
*   **A**：日志查询需要时间，高峰期可能会有延迟，可以等待10秒左右再次点击查看日志。

### 8. 测试的订单如何申请退款？
*   **A**：未使用的订单可直接申请退款，需要在景区系统里操作同意。已核销或不可退的订单，可以汇总联系美团业务同学操作退款，或在订单页面走退款客服申诉。
