# Bug Triage

本技能用于在当前仓库中排查 Bug、异常和行为回归。

适用场景：

- 接口报错
- 编译失败
- 运行时报错
- 业务逻辑与预期不符
- 回归问题定位

## 1. 最小读取路径

先读取：

1. `AGENTS.md`
2. `.agent/architecture/AI_QUICKSTART.md`
3. `.agent/rules/AI_BEHAVIOR_RULES.md`
4. `.agent/architecture/PROJECT_STRUCTURE.md`

然后按问题所在业务域补充：

- `.agent/requirements/{业务域}/`
- `.agent/designs/{业务域}/`
- `.agent/references/{业务域}/`

不要在没有必要时全文重读大型文档。

## 2. 排查原则

- 先确认问题所在模块，不要一开始全仓库扩散搜索
- 先看真实代码和调用链，再看模板文档
- 优先复用现有相似实现和相邻业务逻辑作对照
- 若发现 AI 先前理解与真实代码不符，以真实代码为准

## 3. 标准排查流程

1. 明确症状
2. 定位模块
3. 读取相关业务文档
4. 查接口入口或报错起点
5. 顺着 `Controller -> Service -> Mapper / 外部调用` 往下看
6. 查对象转换、参数构造、枚举、常量和配置
7. 找最近相似实现做对比
8. 做最小修复
9. 做最小验证

## 4. 常见排查入口

优先检查：

- Controller 参数是否正确接收
- DO/DTO/BO/Entity 字段是否对齐
- `Long` / 时间字段序列化和反序列化是否异常
- `QueryDO` 分页和排序参数是否正确
- Mapper 条件是否漏写或误写
- 基类字段是否被重复声明导致映射异常
- 网关、鉴权、租户、逻辑删除是否影响结果
- Nacos / 配置项 / 常量值是否偏离模块现状

## 5. 查代码时的推荐顺序

- 先找报错接口或入口类
- 再找接口实现和调用链
- 再看关联对象模型
- 再看配置、枚举、基础类
- 最后再扩大到跨模块调用

如果已有报错栈，优先围绕栈信息精确定位，不要盲搜。

## 6. 修复原则

- 只改能解释当前问题的最小范围
- 不顺手重构无关代码
- 不改变既有模块风格
- 若发现真实问题来自历史约定或架构边界，应在结果中明确指出

## 7. 验证要求

修复后至少完成一项验证：

- 定向编译相关模块
- 运行相关测试
- 手工检查关键调用链
- 对照相似实现确认行为一致

若无法验证，要明确说明原因和剩余风险。

## 8. 输出要求

结论中应包含：

- 根因
- 修复点
- 影响范围
- 已做验证
- 未验证风险

若问题未完全定位，也要明确已排除的方向和下一步最优检查点。
