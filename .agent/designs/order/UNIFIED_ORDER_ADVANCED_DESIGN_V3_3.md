## 1. 拆单与合单标识约定 (`parent_order_id = 0`)

> 你提到的： “`parent_order_id = 0` 代表主单” 这个约定非常专业。

**为什么不用 `NULL` 而是定死为 `0`？**
1. **索引友好**：在 MySQL (InnoDB) 中，如果你对 `parent_order_id` 这个外键/关联字段打普通索引，包含 `NULL` 的行或者使用 `IS NULL` 的查询在某些版本下会导致索引失效或是极低的过滤效率。而指定默认值为 `0` 并用作条件能稳定走索引。
2. **代码防雷**：在 Java 当中，用包装类 `Long` 或者基本类型 `long` 接参查表时，判断 `== 0L` 的语义比判断 `!= null` 在业务逻辑表达上更明确，更不容易导致 NullPointerException 的连锁反应。
3. **查询收敛**：比如要统计平台整体销量流水，或者是前端查询某人所有的独立大包裹，可以直接硬气地写死条件：`SELECT * FROM o_order WHERE parent_order_id = 0`。

## 2. 并发扣库存方案：Redis Lua 换成 Redisson 

> 你提到的： “Redis Lua 是否可以替换成 Redisson 方案”

**绝对可以，并且强烈推荐替换！**
之前我之所以提到 Lua，核心诉求是**保证“查库存 + 扣库存”的原子性**（不能发生：查到剩1个，但两个线程同时各自扣了1，结果变 -1 超卖）。由于我们项目栈里已经有 Redisson，那手写易碎的 Lua 脚本就变得毫无必要了。

**推荐的 Redisson 落地姿势（由浅入深）**：

1. **做法一：基于 `RLock` (分布式悲观锁)**
   - **思路**：下单时直接 `lock.lock()` 锁住商品或 SKU，查 DB 或 Redis，有货就减，没货报错，最后 `finally { lock.unlock() }`。
   - **点评**：代码最精简傻瓜，但如果某个网红门票（如故宫抢票）并发极高，这种排队式的阻塞会大幅拖垮服务吞吐量。
2. **做法二（极力推荐）：利用 `RAtomicLong` 或 `RSemaphore` (分布式信号量)**
   - **思路**：将各大 SKU 的可用库存预热到 Redisson 的 Semaphore 里。用户下单时，业务不拿锁，而是直接秒级执行 `RSemaphore.tryAcquire()`。
   - **点评**：Redisson 的 Semaphore 底层**也是自动帮你封装了 Lua 脚本发送给 Redis 的**，所以它天然具备了原生的极速性能和严密的原子性。你既免去了手写裸 Lua 的维护苦恼，又达到了非阻塞限流超高并发扣减的惊人体验。

## 3. 定时器与解耦：明确使用 RocketMQ

> 你提到的： “我们的架构用的是 RocketMQ”

那太完美了。针对防线里的“支付超时自动关单（释放库存）”机制，基于 RocketMQ 会比 RabbitMQ（用死信交换机）或手写定时表扫描优雅得多。

**具体落地玩法**：
1. **极低开销的延迟消息（Delay Message）**：在核心引擎插入一条状态为 `PENDING` 的 `o_order` 时，立刻向 RocketMQ 发射一条 15 分钟（业务约定死）的延迟消息。
   - 如果你们用的 RocketMQ 4.x，可以直接 `message.setDelayTimeLevel(xx)` 调用预设的 15 分钟档位。
   - 如果是 RocketMQ 5.x，支持极高精度的任意定时消息：`message.setDeliverTimeMs(System.currentTimeMillis() + 15 * 60 * 1000)`。
2. **消费者自闭环**：15 分钟后，这台订单微服务的下游 Consumer 才真正“看到”这条消息。拿里面的 `order_no` 去查表，如果此时用户还是 `PENDING`，直接执行关单、并反向去 `RSemaphore.release()` 把好不容易抢到的票加回票池中！

## 4. 彻底理解“订单生命周期的单向性”与“退款枚举”

> 你问到：“一旦订单跨越了 PAID 级，绝不能简单的调回 CANCELED…… 单据生命周期的单向性是保障财务对账清晰的核心底线 这句话的意思是什么？订单表的状态，这状态能包含退款中这个枚举吗？”

**是的！订单表的主状态里不仅【可以】，更是【必须】包含 `REFUNDING (退款中)` 和 `REFUNDED (已退款)` 等强烈的逆向售后概念枚举！**

这个单据模型流转之所以被强调“绝不回头”，其背后原因完全是因为现实世界的**真金白银**。

### 举个对账崩溃的例子（反面教材）：
假设小明买票没付款过了 15 分钟，订单显示状态是 **`CANCELED` (交易取消)**。这很正常。
但假设小红买了票，**付了 100 块钱**（此时商户的微信商户号真正进账了 100 元），订单状态是 `PAID (已支付)`。
过了半天她不想要了，在抖音上发起退款。抖音 SPI 通知我们。

如果此时，咱们图省事，直接把小红的订单主状态重新塞回成 **`CANCELED`**（因为她不用这张票了嘛）。
到了月底对账，财务发现：
> 微信侧流水：“小红退走了 100 元，我们商户减了 100 元”。
> 咱们的业务系统库里跑 SQL 一看：“小红的单子叫 `CANCELED` (交易取消)”。

财务拿着账单指着工程师骂：**“在咱们系统里，小明和小红的状态是一模一样的 `CANCELED`！那小明没付钱取消，小红付了钱又退了取消，我在月末系统报表上怎么能分出哪些单子是我真实产生了财务冲正的？！”**

### 正确的解法：闭环的不可逆状态机
为了让每一笔金钱的发丝都有迹可循，单据生命周期必须形成正向与逆向的隔离墙：

1. 未发生的金钱纠葛，随时可以销毁，叫 **取消 (CANCEL)**。
2. 一旦跨越了 `PAID (已支付)` 的红线，钱进来了。之后的任何作废，哪怕这张票她连二维码都没看一次，业务名义上叫“不要了”，财务名义上叫 **“售后/冲正账”**。
3. 此时主表 `o_order` 的 `order_status` 必须强制前进到 **`REFUNDING (退款中)`**。
4. 在此期间，如你参考 SQL 里的 `o_order_return_apply` 退款申请表，会诞生一份实实在在的回查工单。
5. 微信/抖音确认把这 100 块钱物理退回小红的银行卡后，`o_order` 再度前进到 **`REFUNDED (终局结算退款彻底完成)`** 或者 **`PARTIAL_REFUNDED (极度多见的只退了 1 张票)`**。

所以，在核心领域的枚举类中，加入退单枚举、禁止强行逆转支付成功的订单状态（所谓单向性），是保护你们业务系统不被对账部门和财务清算系统挑刺的最顶级底座设计。
