# 订单履约类型规则设计

## 背景

订单支付成功后，不同商品的履约方式并不相同：

- 有的商品支付后需要立即发码/发券
- 有的商品支付后不立即履约，等待人工或外部系统处理

如果继续把“支付成功 = 发码并进入履约”硬编码在订单主流程中，会导致统一订单入口与具体履约形式强耦合。

## 设计目标

1. 让履约类型成为商品规则的一部分，而不是订单主流程硬编码。
2. 让订单支付成功后的履约逻辑通过门面 + 策略路由执行。
3. 在规则未完善前，保留明确的默认回退行为，避免运行时语义不清。

## 规则字段

在 `o_fulfillment_rule` 中配置：

- `scope_type`
- `scope_id`
- `fulfillment_type`

查询按作用域由细到粗回退：

- `SKU -> SPU -> APP -> PROD`

当前仅定义两个取值：

1. `VOUCHER`
   - 支付成功后立即发码/发券
   - 发码完成后推进订单进入 `DELIVERING`

2. `NONE`
   - 支付成功后不立即履约
   - 当前默认不触发发码，订单保持在 `PAID`
   - 后续可再衍生人工履约、实物发货等能力

## 路由规则

履约门面 `OrderFulfillmentFacadeService` 负责统一入口；
门面内部根据订单子项对应的 `o_fulfillment_rule.fulfillment_type` 选择履约策略。

当前策略骨架：

- `VoucherOrderFulfillmentStrategy`
- `NoopOrderFulfillmentStrategy`

## 混单限制

当前阶段 **不支持同一订单内混合多种履约类型**。

约束如下：

- 若同一订单的多个 `OrderItem` 解析出的 `fulfillment_type` 不一致
- 则直接视为不支持的订单结构，拒绝继续履约

原因：

- 当前主订单状态机尚未为“部分 item 发码、部分 item 待后续履约”建模
- 在状态语义未明确前，先禁止混合，避免后续补复杂补丁

## 默认回退

为兼容历史规则数据，当前在以下场景默认回退到 `VOUCHER`：

- `o_fulfillment_rule` 不存在
- `fulfillment_type` 为空

但这只是一种兼容性回退，不代表长期推荐规则。

长期目标是：

- 新商品规则明确配置 `fulfillment_type`
- 履约门面不再依赖隐式默认

## 当前状态语义

在现有状态机不新增状态的前提下：

- `VOUCHER`：支付成功后发码，随后推进到 `DELIVERING`
- `NONE`：支付成功后不立即履约，订单保持在 `PAID`

原因：

- 当前状态机尚未定义“待人工履约/待发货”的独立状态
- 在没有新状态建模前，把 `NONE` 订单强行推进到 `DELIVERING` 会造成语义失真
